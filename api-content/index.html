如今很多的应用程序是“数据密集型”而非“计算密集型”，单纯的硬件能力几乎已经不再是瓶颈，更大的问题通常来自数据量、数据复杂程度和数据变化速度。 而一个数据密集型应用通常也是基于一些常见的基础组件，例如： 数据库：存储数据，方便以后再次访问。 缓存：存储操作代价昂贵的结果，加速下一次访问。 索引：允许用户通过关键词搜索数据，或者按照各种条件过滤。 流处理：将数据持续发送给别的进程来进行异步处理。 批处理：定期地处理大量积累的数据。 为什么这些功能各异的组件都属于“数据系统”的范畴？ 数据的存储和处理组件逐渐交融。 例如作为数据库的Redis常被拿来用作消息队列；作为消息队列的Kafka又提供了类似数据库的持久化保证。 一个应用程序往往需要使用多种组件，程序的代码需要能够将多种组件结合在一起。 例如，如果在应用程序级做缓存，用数据库来持久化数据，那么程序代码需要保持缓存和数据库之间的数据同步，如下图所示。 也就是说，应用程序使用了底层的这些基础组件，构建了一个新的、专用的数据系统，并且这个数据系统对外提供服务（例如暴露API）时会提供各种保证，例如缓存能够保证写时同步等。 那么如何使用这些基础组件来设计并实现一个完整的数据系统，有很多因素需要考虑，往往需要做出各种权衡，这里我们关注最重要的三个因素：可靠性、可扩展性和可维护性。 可靠性 “即使发生了某些错误，系统仍可以继续正常工作” 一个具有可靠性的应用程序应该是怎样的？ 程序具有用户所期待的功能。 可以容忍用户错误使用或不正确地使用程序。 程序性能能够应对典型场景的负载压力和数据量。 程序能防止任何未经授权的访问和滥用。 故障(fault)：指系统中的一个组件不能按照期望的行为运行。 失效(failure)：整个系统无法向用户提供需要的服务。 通常来说，故障是无法完全避免的，因此我们希望能够设计一个能够容忍故障的系统(fault-tolerant)，能够发现并处理故障，避免引发系统失效。 为了能够设计这样的系统，通常开发过程中会刻意地通过一些手段触发故障，然后对各种故障给出合理的处理方式，从而提高系统的容错能力，很多bug就是由于错误处理不当造成的。这样当真实故障发生时，能够正确被处理。 硬件故障 硬件故障是系统故障中很重要的一部分，例如磁盘损坏、电源终端、网络中断等，尤其是当许多硬件同时运行时，这种故障几乎一直在发生。 一个通用的降低硬件故障引发系统失败的方法是硬件冗余(redundancy)。例如将多个硬盘组成RAID阵列、提供备用电源、使用支持热交换的CPU组合等。当其中一个硬件宕机之后，备份的硬件能够立刻顶上，给损坏的硬件更换的时间。实践证明这种方式对于系统的持续运行是很有帮助的。 另外一个途径是在软件层面使用一些容错技术，使得部分硬件故障能够被处理。例如在一个拥有多台机器的系统上，如果需要对机器进行升级，那么相比于全部停止提供服务，还可以逐个地进行升级(rolling update)，保证整个系统还在持续提供服务。 软件错误 通常我们认为硬件故障都是各自独立的，一个机器上的硬盘损坏与另一个机器几乎没有关系。但是软件错误则不同，这些错误更难以察觉（通常在corner case中被发现），并且更倾向于导致整个系统各个部分都出现问题。例如： 一个进程用尽了所有的共享资源，包括CPU、内存、网络带宽等。 程序所依赖的一些服务开始出错、无响应等。 级联故障。一个组件上的小错误导致更多其它组件出现错误。 出现这些软件错误的一个重要的原因是：在开发过程中，对程序运行的环境做出了一些假设，这些假设大多数时候是合理的，但是在极少情况下是错误的。 对于这种错误，没有什么快速的解决方案，但是有很多小的手段可能会有所帮助，例如：开发时认真检查依赖的假设条件、多个进程之间进行隔离、允许进程自动重启、不断地检测系统运行的行为等。 人为错误 在软件周期里，人是最不可靠的因素。为了尽可能消除人为因素对程序造成的负面影响，有以下一些途径： 以最小出错的方式来设计系统。例如精心设计抽象层、API等，使得“做正确的事”更简单。但如果限制过多，人们就会想要绕过它。 设法分离最容易出错的地方、容易引发故障的接口。例如提供一个Sandbox环境进行试用。 充分的测试。从单元测试到集成测试，以及手动测试。 提供快速的恢复机制，来应对人为失误。例如快速回滚配置。 实现监控系统，监控性能指标、异常等。类似其它领域的遥测(Telemetry)。 进行程序的规范化开发管理，并进行人员培训。 可扩展性 即使负载增加，系统也能够正常运转 描述负载 在讨论可扩展性的时候，首先要明确，对于当前的系统，负载到底意味着什么？它通常可以由若干个参数来描述。选择什么参数取决于系统的功能，例如对于服务器来说，每秒请求处理数是很重要的。 以twitter为例，用户发布tweet的频率平均4.6k/s，而用户浏览主页时间线的频率平均300k/s。因此性能瓶颈在于巨大的“扇出(fan-out)”，即每个用户会关注很多人，也会被很多人关注。为了让用户查看时间线时，尽可能缩小延迟，有2种设计方案： 采用关系型数据模型。 存在一个全局的tweets表、一个表示关注关系的follows表，一个用户信息的users表。 用户发送tweet就是向tweets表中添加一行记录。 用户读取时间线时，首先从follows表中获取该用户的所有关注，然后从tweets表中找出所有关注者的tweet，最后按照发送时间排序。 采用数据流水线方式。 同样存在一个全局的tweets表，但每个用户的时间线都会维护成为一个缓存。 用户发送tweet时，先向tweets表中添加，然后会推送到它的每个粉丝的时间线缓存中。 用户读取时间线时，直接从他自己的时间线缓存中读取tweets即可。 第2种方案因为在用户发送tweet时做了更多的事情，因此用户读时间线的性能可以有效得到提升。但是有一个问题是，现实中关注数量的分布是极度不均衡的，对于有1000万粉丝的用户，发送tweet是就会有巨大的写入操作。 后来twitter将这2种方案做了融合，对大部分用户采用方案2，但对粉丝数巨大的用户，仍然采用方案1。 在这个例子中，每个用户粉丝数的分布情况是一个关键的负载参数，因为它决定了扇出数，进而决定了哪种方案拥有更好的性能。 描述性能 当负载增大时，通常有2种思考方式： 若系统资源不变，系统性能会怎样？ 若要性能不变，需要增加多少资源？ 无论如何考虑，都先要能够描述系统的性能。例如，在批处理系统中，吞吐量(throughput)是一个关键的指标；而在线系统中，服务的响应时间(response time)往往很重要。 以响应时间为例，由于各种随机因素的影响，响应时间往往会有一些抖动，例如TCP重传、缺页中断、磁盘IO等。而相比于常用的平均响应时间来说，采用百分位数是更加合适的。例如中位数响应时间(p50)就意味着有一半的请求花费不到该时间。 而在成熟的产品中，也会使用百分位数来描述SLA(Service Level Agreements，服务质量协议)。例如声明说响应时间中位数(p50)小于200ms，而p99小于1s，且至少p999的时间都要达到这个指标。 在使用高百分数响应时间时，延迟排队是一个很重要的影响因素。因为服务器并行的程度有限，因此越靠后的请求，处理得越慢，称为“队头阻塞”。即使后续请求很简单，在客户端也会观察到很长的响应时间。 应对负载增加的方法 那么当负载参数增加的时候，应该如何将性能指标保持良好呢？ 首先，如果实际负载达到了预设值的10倍时，从设计架构的层面上就已经无法应付了。因此每增加一个数量级的负载，需要重新设计架构。 其次，我们需要在垂直扩展(使用更强的机器)和水平扩展(加机器)之间取舍。通常垂直扩展更简单，但是能够应对的负载增加程度有限，往往还是要进行水平扩展。对水平扩展来说，如果扩展的是无状态服务(stateless)会比较简单，如果扩展的是有状态(stateful)服务会更复杂。 对于超大系统来说，往往是对特定应用定制的，很难有通用的架构。通常需要在读取写入量、访问模式等中间做取舍。例如100k/s的1KB大小的请求，和3/s的2GB大小的请求，即使吞吐率相同，架构也会大相径庭。 可扩展的架构通常是由通用模块逐步构建得到，通过这些通用模块和常见的架构模式进行研究，对设计可扩展架构是很有帮助的。 可维护性 软件开发的大部分成本，都花在投入运行的阶段，包括维护和故障修复、适配新的平台和场景、增加新功能等。 为了尽可能减少维护期间的麻烦，在设计的时候就应该开始考虑，避免造出容易过期的系统。有三个原则需要关注： 可运维性：方便后续的运维。 简单性：简化系统内部实现的复杂性。 可演化性：使得后续工程师也能够轻松改进系统，并适配到非典型场景中，也称为可延伸性、易修改性、可塑性。 可运维性 良好的可操作性能够有效化解软件的局限性。而在数据系统设计时就考虑到可操作性，是很有必要的。 例如提供对系统运行时行为和内部指标的监测、易出错的操作尽量自动化、良好的文档和操作模式、出现故障时能够尝试自我修复等。 简单性 当项目越来越大时，很容易变得复杂。例如状态空间发生膨胀、模块紧耦合、令人纠结的相互依赖、为了性能采取的特殊处理等。 简化系统设计并不是指减少系统的功能，而是“消除意外方面的复杂性”，即并非软件固有或者被用户感知，而是实现本身衍生出来的问题。 消除这种意外复杂性最好的方法是抽象。一个好的设计抽象可以隐藏大量的细节，并提供干净易懂的结构。而且好的抽象往往更容易被各种不同程序复用。 例如高级编程语言就是一种抽象，隐藏了机器汇编代码、CPU寄存器和系统调用等细节；SQL语言这种抽象隐藏了内部复杂的磁盘和内存数据结构，以及多客户端并发请求。 通过研究如何设计好的抽象，至少能够将大型系统的一部分抽象为定义明确、可重用的组件，帮助系统的复杂度降低到可控的级别，是很有意义的。 可演化性 可演化性是指数据系统级的敏捷性，即当面对一个由多个不同特性的应用或者服务协作构成的大型系统时，也能够很方便地实现新的系统需求。 小结 ","link":"https://pikameow.github.io/post/DDIA-Chapter1/"}]}
