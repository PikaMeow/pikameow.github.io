词法分析 -> 语法分析 -> 语义分析

首先是词法分析，对于这道题而言，真正需要状态转换的只有三种元素：变量名(由字母和数字组成)，数(由数字和小数点组成)，空白符(包括空格、换行符、制表符等)。其余的都是单个字符，一步转换就可以到达接受态。而上述的三种中每一种的状态数也都很少，所以可以直接根据正则表达式做出DFA来，如图：

然后就是语法分析，这里用的是LL(1)文法，大致分为以下几步

  1. 写出产生式
  2. 消除左递归和左因子
  3. 求出每一个非终结符的FIRST和FOLLOW集合
  4. 确认是否为LL(1)文法(若不是要重新设计产生式)
  5. 根据两集合做出LL(1)转换表
  6. 用栈或者递归的方式实现语法分析

首先，FOLLOW集合记录的是可能跟在某个非终结符之后的终结符的集合，仔细思考整个算法的执行过程，可以发现这样几个问题：

为什么要从开始符号SSS开始寻找FOLLOW集合中的元素？实际上没有硬性的要求，从第4步的描述我们可以看出来，整个算法的执行次数是不确定的，它由产生式的数量和复杂程度，以及每个非终结符对应的FIRST集合的大小来决定。只是因为开始符号代表着整个输入集合，而这个集合必然以$结尾，所以我们先把它加进去，主要是出于算法逻辑方面的考虑。

第2步和第3步的意义在哪里？对于第2步来说，对于产生式右边的某个非终结符，如果在它的右边还有符号的存在(包括非终结符和终结符)，那么这个符号的FIRST集合中的所有元素都可能出现在那个非终结符的后面，当然，如果这个符号是ϵ\epsilonϵ，相当于不存在这个字符，也就不按照这个规则处理，而按照第3步来处理。第3步告诉我们，对于产生式A→αBA\to \alpha BA→αB,如果产生式右边的非终结符BBB后面没有符号了，那么所有可能出现在AAA后面的字符，也都有可能出现在BBB的后面。这两步作为一个整体，可以保证对于每一个通过第2步改变FOLLOW集合的BBB，都会在算法条件满足之前，以位于产生式左端的身份通过第3步改变它所产生的某些非终结符的FOLLOW集合的大小。

为什么第3步不是FOLLOW(A)=FOLLOW(A)∪FOLLOW(B)FOLLOW(A)=FOLLOW(A)\cup FOLLOW(B)FOLLOW(A)=FOLLOW(A)∪FOLLOW(B)呢？实际上对于这种情况而言，如果将AAA和BBB的FOLLOW集合确定下来，可以发现，FOLLOW(A)=FOLLOW(B)FOLLOW(A)=FOLLOW(B)FOLLOW(A)=FOLLOW(B)，对于每一个非终结符，如果它位于某个产生式的右边，那么它一定位于另一个产生式的左边(因为所有的非终结符都是终结符的集合)；反之也成立，但是要除去起始符号SSS.一般而言，某个非终结符会先位于某个产生式的右边，才再位于另一个产生式的左边，因此我们只不断增大位于产生式右端的FOLLOW集合，就可以保证顾及到了所有的非终结符。如果第3步改成题面上的这种形式，某个产生式BBB通过第二步改变了FOLLOW集合，但是当它作为产生式头的时候，就不会再去改变它所产生的非终结符的FOLLOW集合，而且无论循环多少次都是这样。比如有A→BA\to BA→B,B→numB\to numB→num，BBB只出现在这两个产生式里，那么算法执行完毕,BBB中的元素一定为空。

至于具体的实现的两种方式，一种是栈,一种是递归，当然都是基于LL(1)表的，我在做的过程中，一开始使用了栈实现，没有显式地构造语法树，想着在语法分析的时候直接进行语义分析，但是实现到最后还是出了一些问题，就是无法在建树的过程中实现对树的前序遍历，导致对非终结符属性的赋值顺序出现了问题，而且在语义分析时基本没有用到栈里的信息，或者说不知道该怎么用...于是就放弃了，改用递归实现，可以比较清楚地把相应的语义分析的语句加到合适的位置上，然后就搞定啦~

其中维护了一个变量的表，指示了变量的信息，包括类型、名字、值及是否被赋值，在执行声明、赋值、输出语句的时候都要跟这个表打交道。

最后是错误处理部分，分为词法错误、语法错误和语义错误。词法错误说明是某个输入符号的问题，指明出错的具体位置(所处的行和列)；语法错误说明是某个token的问题，要指出token的类型和对应的值；语义错误主要处理变量是否声明和赋值，以及是否重复声明的过程。而且发现如果前面按照正常的流程进行分析，最后进行错误处理时只用改动很少部分代码，而且很容易修改。
